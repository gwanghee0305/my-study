###### **1. 관계형 데이터베이스 (RDB)**

&#x09;RDB(Relational Database)는 데이터를 행(Row)과 열(Column)로 구성된 2차원 표(Table) 형태로 관리하는 데이터베이스입니다.

&#x20;	데이터 간의 고유한 \*\*'관계(Relationship)'\*\*를 맺어 데이터를 관리하며, 데이터를 조작할 때 표준 SQL을 사용합니다.



**ACID : 트랜잭션의 무결성 보장 특성**



원자성 (Atomicity)    -------	작업은 전부 성공하거나 전부 실패, 중간 상태 없음



일관성 (Consistency) -------	트랜잭션 후에도 데이터의 유효성이 항상 유지됨



고립성 (Isolation)	     -------	동시에 실행되는 트랜잭션이 서로 영향을 주지 않음



지속성 (Durability)    -------	커밋된 데이터는 시스템 장애가 발생해도 영구 보존



EX)  A가 B에게 10만 원을 송금할 때:



&#x20;\*\*Atomicity	: 출금과 입금이 동시에 성공하거나, 둘 다 취소됨

&#x20;\*\*Consistency	: 이체 전후 총 금액은 항상 같아야 함

&#x20;\*\*Isolation		: 다른 사람의 동시 이체가 이 거래에 영향을 주지 않음

&#x20;\*\*Durability	: 완료 후 서버가 꺼져도 데이터는 보존됨

###### **2. NoSQL (Not Only SQL)**

&#x09;NoSQL은 말 그대로 "SQL뿐만 아니라 다른 것도 지원한다"는 뜻으로, 전통적인 RDB의 형태(표 구조)의 한계를 극복하고자 

&#x09;등장한 비관계형 데이터베이스를 의미합니다.



&#x20;	1)  고정된 테이블 구조 없이 ""유연한 형식""으로 데이터를 저장합니다.



&#x09;2)  ""대용량 데이터 처리""와 수평 확장(Scale-Out)에 강합니다.



&#x20;	3)  스키마가 없거나 느슨하게 정의되어 ""빠르게 변화하는 데이터 구조""에 적합합니다.



&#x20;	4)  ""속도·유연성·확장성""을 최우선으로 고려합니다.



**NoSQL 대표 유형**



\*\* 유형 \*\*

1\) 문서형 (Document)	

2\) 키-값형 (Key-Value)	

3\) 열 지향 (Column-Family)

4\) 그래프형 (Graph)	



\*\* 저장 방식 \*\*

1\) JSON/BSON 형태의 문서 저장

2\) Key로 Value에 빠르게 접근

3\) 열 단위로 저장·조회

4\) 노드-간선 구조로 관계 표현



\*\* 활용 예시 \*\* 

1\) 블로그 포스트, 상품 정보

2\) 세션 관리, 캐시

3\) 로그 수집, 시계열 데이터

4\) 친구 관계, 추천 시스템

===============================================================================

###### 3\. 관계형 데이터베이스와 NoSQL 차이점



**1) 데이터 구조 및 스키마 (Schema)**

&#x20; 1-1) RDB (Relational Database)

&#x20;  	- 고정된 스키마: 데이터를 저장하기 전에 테이블의 구조(컬럼 이름, 데이터 타입 등)를 엄격하게 정의해야 합니다.

&#x20;  	- 정형 데이터: 모든 행(Row)은 반드시 동일한 구조를 가져야 하며, 빈 값은 NULL로 처리됩니다.

&#x20; 1-2) NoSQL (Not Only SQL)

&#x20;  	- 유연한 스키마: 정해진 틀이 없습니다. 데이터 모델에 따라 Key-Value, Document(JSON 형태), Graph 등 

&#x09;  다양한 방식으로 저장합니다.

&#x20;  	- 반정형/비정형 데이터: 같은 컬럼(필드)이 없어도 상관없고, 필요에 따라 언제든 새로운 필드를 추가할 수 있습니다.



**2) 데이터 일관성과 트랜잭션 (ACID vs BASE)**

&#x20;2-1) RDB

&#x20;  	- ACID 특성 준수: 데이터의 정확성과 무결성을 최우선으로 합니다. 환경에서 완벽한 트랜잭션(원자성, 일관성, 고립성, 지속성)을 	 보장합니다.

&#x20;2-2) NoSQL

&#x20;  	- BASE 특성 따름: 일관성보다는 가용성과 분산 처리 속도를 중시합니다.

&#x20;  	- 데이터가 업데이트되었을 때 모든 서버에 즉시 반영되지 않더라도, 결국에는 시간이 지나며 싱크가 맞아 떨어지는 방식

&#x09; (결과적 일관성)을 취합니다.



&#x20; **※** RDB는 데이터 구조가 명확하고, 변경될 일이 적으며, 정확성이 핵심이며,

&#x20;    NoSQL은 데이터 구조가 자주 바뀌고, 대규모의 데이터를 빠르게 읽고 써야 할 때 선택
